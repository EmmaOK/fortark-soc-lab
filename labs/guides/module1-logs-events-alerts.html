Logs, Events, and Alerts

20 minutes
Foundational
Read before Lab 1

The server's diary

Every action that happens on a computer leaves a trace. A user logs in — logged. An application starts — logged. A file is read — optionally logged. A login fails — logged.

These traces are called logs. They are timestamped records of things that happened, written automatically by the operating system and every major application running on it.

Here is a real log entry from an SSH server being brute forced:

May 30 13:38:58 lab-target sshd[12341]: Invalid user admin from 140.82.114.4 port 45231

Let us break it down:

Part Value What it means
Timestamp May 30 13:38:58 When it happened
Hostname lab-target Which machine it happened on
Process sshd[12341] Which program wrote the log (SSH daemon, process ID 12341)
Message Invalid user admin What happened — someone tried to log in as "admin" but that account does not exist
Source IP 140.82.114.4 Where the attempt came from
Source port port 45231 The network port the attacker used

One line. One failed login attempt. Seventeen fields of information about exactly what happened, when, where it came from, and why it failed.

Now imagine 200 of those lines arriving in the space of 60 seconds.

The scale problem

A typical Linux server running a web application generates between 50,000 and 500,000 log lines per day. A medium-sized organisation with 500 servers generates tens of millions of log lines daily.

No human being can read that. No team of humans can read that.

This is the fundamental problem that a SIEM solves.

What is a SIEM?

SIEM stands for Security Information and Event Management. It is a system that:

  1. Collects logs from every server, application, and network device in the organisation — centralising them in one place
  2. Parses each log line — extracting the structured fields (timestamp, source IP, username, etc.) from the raw text
  3. Applies rules — comparing what is happening against known patterns of attacks and anomalies
  4. Fires alerts — when a rule matches, creating a structured security event that an analyst can investigate

Think of it this way: if logs are the raw diary entries written by thousands of servers simultaneously, the SIEM is the analyst who reads all of them, recognises danger patterns, and taps you on the shoulder when something needs your attention.

In this course, the SIEM is Wazuh. It is collecting logs from your student target (s[N]-lab-target) in real time. Every SSH login attempt, every file change, every user action generates a log line — Wazuh receives it within seconds and checks it against its ruleset.

What is a rule?

A rule is a condition that, when met, generates an alert.

Rules can be simple or complex:

Simple rule:
If an SSH login fails, fire an alert at level 3.
Correlation rule:
If more than 8 SSH login failures occur from the same source IP within 120 seconds, fire an alert at level 10.

The second type — a correlation rule — is where a SIEM becomes genuinely powerful. It is not just recording individual events. It is recognising that eight failed logins in two minutes from the same IP address is a pattern with a specific meaning: someone is running an automated attack.

This is exactly what Wazuh rule 2502 does. It does not care about any single failed login. It watches the aggregate — and when the pattern crosses a threshold, it tells you.

The alert hierarchy

It helps to understand the difference between a log, an event, and an alert:

Raw log line
"May 30 13:38:58 lab-target sshd[12341]: Invalid user admin from 140.82.114.4"
        ↓
    Parsed event
    { timestamp, source_ip, target_user, action: "login_failure", reason: "invalid_user" }
        ↓
    Rule match → Alert
    Rule 5710 fires: "sshd: Attempt to login using a non-existent user" — Level 5
        ↓
    Correlation → Higher alert
    Rule 2502 fires: "syslog: User missed the password more than one time" — Level 10
        ↓
    Correlation with success → Critical alert
    Rule 40112 fires: "Multiple authentication failures followed by a success" — Level 12

Each step adds meaning. The raw log line tells you one thing happened. The correlation rule tells you what it means. Rule 40112 at the bottom of that chain is not just an alert — it is a conclusion: someone got in.

Three rules you will see in Lab 1

Rule What triggers it Level What it means
5710 Single failed SSH login for a non-existent user 5 One attempt. Noise by itself.
2502 8+ failures from the same IP in 120 seconds 10 Brute force pattern confirmed.
40112 Multiple failures from an IP followed by a successful login 12 Breach confirmed.

Notice the level progression: 5 → 10 → 12. Wazuh is not just detecting events — it is escalating its confidence as the evidence accumulates.

A level 5 alert says: "Something happened. FYI."
A level 10 alert says: "This is a pattern. Pay attention."
A level 12 alert says: "This is an incident. Act now."

What "Invalid user" vs "Failed password" actually means

This distinction will come up in Q3 of Lab 1 and it is one of the most practically important things in this module.

When an attacker tries to log in as a username that does not exist on the machine, the log says:

Invalid user admin from 140.82.114.4

When an attacker tries to log in as a username that does exist but gives the wrong password, the log says:

Failed password for root from 140.82.114.4

Both look like "failed login." Both generate similar alerts. But the second one is significantly more dangerous — it means the attacker has already identified a valid account. They just need to find the password.

In Lab 1, hydra targets the username admin. That account does not exist on the target machine. Every attempt generates "Invalid user admin" — level 5 noise. Then you manually log in as victim (a real account) — and rule 40112 fires at level 12. That escalation is not accidental. It reflects the actual change in risk.

Why centralising logs matters

Before SIEMs, every server kept its own logs locally. An analyst investigating an attack had to SSH into each individual server, read logs manually, and try to piece together a timeline across dozens of machines.

Attackers exploited this. After a breach, one of the first things an attacker does is delete or modify the local logs on the machine they compromised. Evidence gone.

With a SIEM, logs are shipped to a central server immediately — before an attacker can delete them. By the time the attacker thinks about covering their tracks, the evidence is already somewhere else.

This is why Wazuh is running as a separate system from your student target. Even if the lab-target machine were completely destroyed, Wazuh already has the evidence.

Key concepts from this module

  • Logs are timestamped records of system events — every server generates thousands per day
  • A SIEM collects, parses, and analyses logs centrally — no human can read raw logs at scale
  • Rules define patterns; alerts fire when patterns match — moving from "something happened" to "this means something"
  • Correlation rules connect multiple events into a single meaningful alert
  • "Invalid user" and "failed password" are not the same thing — one means no valid username, the other means the attacker has a valid username and needs only the password
  • Centralised logging prevents attackers from erasing evidence after a breach
