What is a supply chain attack?

Two terms you need to understand first
Supply Chain
Every application depends on dozens — sometimes hundreds — of external libraries written by other people. This web of dependencies is the supply chain. When a developer runs pip install something, they are trusting everyone who wrote that library and everyone who has ever had access to publish it. A supply chain attack hides malicious code inside one of those trusted packages.
Post-Install Hook
A script that runs automatically when a software package is installed. Normally used for legitimate setup steps. In a supply chain attack, the attacker adds a malicious step here: write a file to disk, make an outbound network call, install a cron job — all silently, inside what looks like a perfectly normal install log, while the developer watches and sees nothing unusual.

Analogy: Imagine a restaurant that gets vegetables delivered every morning. One day, a supplier's warehouse is secretly compromised. The crates still look the same, the labels are correct, and the food tastes fine. But hidden inside one crate is a small device that unlocks the restaurant's back door. The manager never checked inside the crates — why would they? They trusted the supplier. That trust is the vulnerability.

The scenario

Nobody ran Hydra this time. Nobody brute-forced a password. The attacker never directly touched your server.

A developer on your team installed a Python package three days ago. It looked legitimate — a popular data utilities library with thousands of downloads. They ran pip install fortark-utils==2.1.4 during a routine dependency update. The terminal showed a normal install log. Everything looked fine.

What the developer did not see: three weeks before this install, the original author's PyPI account was compromised. The attacker published version 2.1.4 — a version that did not exist before. The package still worked exactly as advertised. But the setup.py post-install hook ran an extra step: it silently wrote a file to disk, executed it, phoned home, and installed a cron entry that will run every 5 minutes until someone finds and removes it.

This is a supply chain attack. The attacker never needed credentials, never triggered a failed login, never appeared in your auth logs. They arrived in a legitimate package, carried by a legitimate user, from a legitimate source.

Your dashboard is showing three alerts that fired within 2 minutes of each other. None crossed your high-severity threshold. None are individually alarming. Together, they tell a completely different story.

Q7
The FIM alert shows this file as "new file created." Why would a junior analyst dismiss this alert on its own?
New files in /tmp/ are among the most common FIM events — build tools, package managers, compilers, and IDEs all create temporary files there constantly. A single low-severity alert for a new Python file would be lost in the noise. The signal only becomes visible when you notice it arrived at the same time as the other alerts.

What User-Agent did the callback use, and why does that matter?
The User-Agent is pip/23.0 — making the outbound HTTPS POST look identical to a normal pip package manager request. Most firewalls and proxy logs show this as routine package management. An analyst reviewing network logs without context would classify it as noise.
Q10
Is httpbin.org flagged by any threat intelligence feed? What does that tell you?
httpbin.org is a legitimate developer testing service — clean on AbuseIPDB, VirusTotal, and Shodan. Attackers deliberately use legitimate infrastructure because these domains are whitelisted on nearly every corporate firewall. This is why IP reputation cannot catch this class of attack. Behavioural detection — watching what a host does, not where it sends traffic — is the only approach that works.

Q13
Why is the persistence stage often the last alert to fire — and the most critical to find?
File drops and network callbacks happen immediately on install. The cron entry creates a repeating beacon that survives reboots, survives deletion of the original payload file (the cron reinstalls it), and continues calling back to the attacker every 5 minutes indefinitely. In real incidents, the cron entry is often discovered days or weeks after the initial infection — long after the original alerts were forgotten.

Q17
A junior analyst says: "httpbin.org is clean, so the callback wasn't malicious." What is wrong with that reasoning?
The destination being clean does not mean the traffic is benign. The malicious behaviour is: a Python file that was silently dropped by a package post-install hook is beaconing system information (hostname, username, OS version) to an external server. The choice of destination is irrelevant to the malicious intent. Attackers choose clean destinations precisely because analysts make this mistake.

Part 6 — The SOAR gap

Lab 3 connection — why SOAR did not catch this

Lab 3 showed SOAR automating case creation for brute force attacks using rule 40112 — which requires a failed login sequence followed by a successful authentication. Supply chain attacks produce zero authentication events. The attacker never logged in. The SOAR automation from Lab 3 would never fire.

SOAR automations only catch what they are explicitly programmed to detect — and most rules are written for known attack patterns. Supply chain attacks exploit the gaps between those patterns.

Q18
Why would the Lab 3 SOAR automation NOT have caught this attack?
Q19
What Wazuh alert pattern could you use to build a SOAR automation that would catch this type of attack in the future?
A composite rule that fires when: (1) a new Python or shell script appears in /tmp/ AND (2) within 60 seconds, that file is executed AND (3) within 60 seconds, there is an outbound POST request from the same host. None of these three events individually triggers high severity. The correlation across all three, within a time window, is the trigger. Writing effective correlation rules is one of the hardest ongoing problems in detection engineering — it requires understanding the attack before you can define the rule that catches it.

Key takeaways

  • Supply chain attacks bypass every perimeter control. The attacker never needed a password, never triggered a failed login, never appeared in auth logs. They arrived in a legitimate package carried by a legitimate user from a legitimate registry.
  • Three low-severity alerts can equal one critical incident. No individual alert in this lab crossed the high-severity threshold. The file drop, the callback, and the cron modification each looked routine in isolation. The attack is only visible when you look at all three together, in the same time window, on the same host.
  • IP reputation cannot catch this class of attack. The beacon went to httpbin.org — a clean, legitimate service. Attackers deliberately use trusted infrastructure because analysts rely on blocklists. Behavioural detection is the only approach that works.
  • Persistence is the goal, not the initial infection. The payload drop and callback are one-time events. The cron entry gives the attacker ongoing access. Removing the payload file without removing the cron entry solves nothing — the cron reinstalls it.
  • SOAR automation has blind spots. The Lab 3 automation caught brute force. It would never fire here. SOAR rules are written for known attack patterns. Supply chain attacks are effective precisely because they exploit the gaps between those patterns.
  • The developer was not at fault. The package looked legitimate, came from a trusted registry, and installed without errors. The defence is verification: package signing, hash pinning, and Software Bill of Materials (SBOM) — tools that let you verify what you are installing before you install it.
