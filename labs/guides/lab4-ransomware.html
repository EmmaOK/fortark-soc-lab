What is ransomware?

Two terms you need to understand first
Ransomware
Malicious software that encrypts your files — scrambles them mathematically so they cannot be opened — then demands a payment for the key to unscramble them. Encryption itself is legitimate technology. Ransomware weaponises it: the attacker holds the decryption key, you hold encrypted files you can no longer read. Without backups, you pay or you lose the data permanently.
FIM — File Integrity Monitoring
A security control that watches files and directories and fires an alert whenever something changes — a file is created, renamed, modified, or deleted. Against ransomware, FIM becomes the primary detector: when ransomware renames hundreds of files from .docx to .locked in under 10 seconds, FIM fires hundreds of alerts in rapid succession. That storm is the signature.

The scenario

The attacker from Labs 1–3 did not leave empty-handed. Before exiting, they planted a ransomware payload — a dormant script timed to activate when the target's defenses are quiet.

You are opening your morning shift. Your SOC dashboard starts showing something unusual: a flood of File Integrity Monitoring alerts, all from the same host, all inside a 30-second window. Files are being deleted. New files are appearing with extensions you don't recognise — .locked where there used to be .docx, .csv, .sql. And then you see it in /tmp/:

HOW_TO_DECRYPT.txt

The previous labs asked you to detect and document. This lab asks you to do something harder: decide. The encryption may still be running. Every second of delay is another file gone.

How ransomware is detected

Ransomware leaves a signature across five phases. Each phase is visible in your SOC tooling — but only if you know where to look:

Stage What the ransomware does Where it shows up
1 Discovery Scans filesystem for target files Wazuh syslog STAGE:DISCOVERY tag
2 Staging Creates victim file set in working directory Wazuh FIM Rule 554 — files added
3 Encryption Renames every file to .locked Wazuh FIM Rule 553 + 554 — delete + add per file
4 Ransom note Drops HOW_TO_DECRYPT.txt in each directory Wazuh FIM Rule 554 — new file in monitored path
5 Inhibit Sweeps for backup files to delete Wazuh syslog STAGE:INHIBIT tag

The key signature is not any single alert — it is the pattern: dozens of FIM events from the same host in a tight time window, alternating between deletions and additions, ending with a ransom note filename.

Part 1 — Understand File Integrity Monitoring

Before you trigger the attack, understand the detection mechanism so the alerts make sense when they flood in.

What FIM does

Wazuh watches specific directories on each host. Every file in those directories has a recorded checksum. When a file is created, renamed, or deleted, Wazuh fires an alert within seconds.

In a normal environment, a monitored directory might change a handful of times per day. Ransomware changes dozens of files per minute. That volume spike is the signature.

Three FIM rules to know

RuleMeaningWhat ransomware triggers it
554 File added to monitored directory Every .locked file created; ransom note created
553 File deleted from monitored directory Every original file removed after encryption
550 File modified (content changed) Files modified in-place during encryption

When ransomware renames 40 files to .locked, Wazuh sees two events per file: the original disappears (rule 553) and a new file appears (rule 554). 40 files = 80 FIM events.

Before you run anything, open Wazuh and run this filter with time range Last 15 minutes:

agent.name: s[N]-lab-target AND rule.groups: syscheck
Q1
How many FIM alerts does your target show right now, before the simulation?
That quiet baseline — zero to a few alerts — is what makes the ransomware spike immediately obvious. When you return to this filter after triggering the simulation, the contrast will be striking.

Part 3 — Hunt the attack stages

The simulation logged five distinct stages. Your job is to find evidence of each one in Wazuh — as if you had no idea what the script did.

🔍
Stage 1
Discovery
T1083

The ransomware first scans the filesystem to find files worth encrypting. This is logged via syslog and picked up by Wazuh. Go to Threat Hunting → Events (not FIM — no files have changed yet at this stage).

In the search bar, type the keyword you expect to see in the log and press Enter:

"STAGE:DISCOVERY"

Click the matching event to expand it. You will see the full alert JSON — read through the key fields:

  • full_log — the raw syslog line, showing exactly what the ransomware logged
  • rule.id — the Wazuh rule that matched (100400)
  • rule.description — human-readable explanation of what fired
  • rule.mitre.technique — the ATT&CK technique (T1083 — File and Directory Discovery)
  • rule.mitre.tactic — the ATT&CK tactic (Discovery)
  • predecoder.program_name — confirms the source process (lab4-ransomware)

Once you find an event this way, note the rule ID. For future filtering you can use rule.id: 100400 directly — but always start with a keyword search so you understand why that rule ID is the right one.

Q3
What directories did the ransomware scan during discovery? (Hint: look at full_log)
Q4
Why does ransomware bother with discovery before encrypting?
Ransomware operators want to maximise the ransom. Encrypting system files breaks the OS — the victim can't pay from a machine that won't boot. Discovery lets the ransomware target only valuable data: documents, databases, spreadsheets. This is why Windows ransomware targets C:\Users\ and mapped network drives, not C:\Windows\System32.

Stage 5
Inhibit Recovery
T1490

After encrypting, the ransomware sweeps for backup files and attempts to destroy them. Start with the sweep trigger in Threat Hunting → Events:

"STAGE:INHIBIT"

This confirms the sweep started. Now search for what it actually found:

"INHIBIT:FOUND_BACKUP"

You will get one event per backup file the ransomware located. Each event's full_log looks like this:

Jun  8 10:27:49 s1-lab-target lab4-ransomware: INHIBIT:FOUND_BACKUP /tmp/lab2-post-exploit/loot.tar.gz

The file path at the end tells you the backup that was targeted. Look at the extensions across all the events — each unique extension represents a backup type the ransomware was programmed to destroy.

Q11
What backup file extensions did the ransomware target? List them from the INHIBIT:FOUND_BACKUP events in Wazuh.
Q12
Why is destroying backups the most dangerous part of a ransomware attack?
Encrypted files without backups mean the only path to recovery is paying the ransom — which funds the attacker's next campaign and has no guarantee of working. Ransomware operators learned this quickly: organizations with good backups don't pay. This is why modern ransomware always targets backup infrastructure first. Offline, air-gapped backups — physically disconnected from any network — are the gold standard.

Part 5 — The containment decision

This is the hardest moment in a ransomware incident. Isolating the host stops the ransomware from spreading — but also ends your ability to watch it phone home to a C2 server, potentially destroying evidence.

The correct decision depends on whether the ransomware is still running. Answer these questions first:

Q16 — Is the encryption still actively running?
Q17
Does this host have access to shared network drives or other servers?
In this lab environment: no — student targets are isolated containers. In a real environment this is the critical question. A compromised host with write access to a shared file server can encrypt thousands of additional files without the ransomware ever touching those servers directly.
Q18 — Your containment decision
Q19
In a real production environment, what command would you run to isolate a Linux host?
Block all traffic except your management connection using iptables: 
iptables -I INPUT  1 -s <YOUR_SOC_IP> -j ACCEPT
iptables -I OUTPUT 1 -d <YOUR_SOC_IP> -j ACCEPT
iptables -A INPUT  -j DROP
iptables -A OUTPUT -j DROP

Or — if the host is a VM — pause it at the hypervisor level. That preserves memory state for forensics while stopping all network activity instantly.

Part 7 — Recovery planning

Detection and containment are the first half. Getting back to operational is the second half — and it takes longer.

1
Do you have clean backups?

Before anything else: when was the last backup taken? Are backups stored on a network share? (If yes — they may also be encrypted.) How long will restoration take?

In this lab, the victim files were generated by the simulation script. There are no backups. The realistic answer: those files are gone. Pay or rebuild from scratch.

2
Can you identify the initial access vector?

You can. The full kill chain for this curriculum:

Enforcing SSH key authentication — disabling password login — would have broken the chain at Step 1 and prevented everything that follows.

The attacker in Lab 2 exfiltrated /etc/passwd (full user list) and the SSH key directory listing. Every user account on this host must be treated as compromised. Before this server returns to production: all credentials rotated, all SSH keys regenerated, all trusted hosts reviewed.

Key takeaways

  • Volume is the signature. A single file change is noise. Eighty FIM events from the same host in 5 seconds is a pattern — and recognising patterns is the core analyst skill.
  • Ransomware detection is a race. The 40 files in this simulation were gone in 2 seconds. Automated detection — not manual log review — is the only way to respond before the damage is done.
  • Backups are the only reliable recovery. Decryption without the key is effectively impossible. Paying the ransom is a gamble, not a recovery plan. Air-gapped, tested, offline backups determine whether this is a 2-hour incident or a month-long crisis.
  • The kill chain started in Lab 1. The ransomware was only possible because the SSH brute force succeeded. Every defence layer that could have stopped it — rate limiting, key-only SSH, network segmentation — would have broken the chain before this lab's events were possible.
  • SOAR helped but did not stop it. Lab 3 created an automated case for the brute force. But the payload was already planted. Automation is not a substitute for fully investigating the initial alert — that case should have triggered a host inspection that would have found the payload before activation.
  • Ransomware operators run businesses. The ransom note includes a wallet, a contact, a file count, a timestamp, and a victim ID. These are optimised through iteration. Understanding the operator's business model helps you predict their next move and find their infrastructure.
