What is C2 beaconing?

Two terms you need to understand first
C2 — Command and Control
A communication channel between an attacker and a compromised machine. After breaking in, the attacker does not stay connected — they set up a channel that lets them send commands and receive results on demand. C2 is how the attacker controls a machine they already own.
Beaconing
The compromised machine regularly "checking in" with the attacker's server — like a heartbeat. Every N seconds, it says: "I'm here. I'm alive. Send me instructions." The attacker reads the check-in and can reply with a command to run, a file to download, or nothing at all.

Analogy: Imagine a spy who calls their handler every 20 minutes from a payphone. Each call is brief — "still here, nothing to report." The handler can say "do nothing" or "go to the embassy now." The spy does not stay on the line. They hang up, wait, and call again. Anyone watching sees a person making a routine phone call. Only someone tracking all the calls over time sees the pattern.

In this lab, the "spy" is a curl process running from a cron job installed by the Lab 5 supply chain attack. The "handler" is httpbin.org. The call happens every ~20 seconds. Your job is to notice the pattern before the attacker sends their first real command.

The scenario

The cron job from Lab 5 is running. Every 20 seconds, a process on s[N]-lab-target wakes up, makes an HTTPS GET to an external host, and goes quiet again. Twenty seconds later, it happens again.

No files are being encrypted. No credentials are being stolen. No ransom note has appeared. Every individual request looks like a completely normal HTTPS connection. The User-Agent says Mozilla/5.0 (compatible; SystemUpdater/...). The destination is httpbin.org. Clean. Routine. Invisible.

Except it has been happening every 20 seconds for the past 8 hours.

This is Command and Control beaconing. The compromised host is checking in with the attacker's server — waiting for instructions. The attacker is not in a hurry. They have time. They are already inside.

Q7
Are the intervals perfectly regular? What does the variation tell you about the implant's design?
The jitter is the tell. A normal software update check fires once a session. A cloud sync client might fire every few minutes. But an application that fires every ~20 seconds with a consistent interval distribution — no matter what else is happening on the system — is following a programmed loop, not responding to user activity.

Part 4 — Find the beacon in syslog

The beacon script logs its activity to syslog at each iteration. This gives you a second corroborating source — and reveals the implant's internal timing, not just when the network packet arrived.

agent.name: s[N]-lab-target AND full_log: *lab6-c2-beacon*

(Look at the BEACON:N lines — what fields are logged?)
Q9
Does the next_in field in syslog match the actual interval you calculated from Suricata timestamps?
The syslog shows what the implant planned to wait. The Suricata timestamps show when the packet actually arrived. Comparing these two reveals the jitter: the implant chose a random value within its jitter range and slept exactly that long. This is the same internal timing you would see in a Cobalt Strike or Metasploit Meterpreter beacon log.

What information is encoded in the User-Agent?
Q11
In a real C2 implant, why would the attacker encode victim data in the User-Agent rather than in the request body?
The request body of an HTTPS connection is encrypted — Suricata cannot read it without TLS inspection. But many proxy servers and SIEMs log User-Agent headers. Embedding victim data in the UA is a trade-off the attacker has already calculated: the data is visible to a sophisticated defender who performs TLS termination, but it ensures the C2 server gets the victim ID even when the response is incomplete.

Q12
What real-world C2 framework does this beacon pattern most closely resemble?
Fixed-interval + jitter, HTTPS to a clean destination, victim data encoded in headers, legitimate tool as transport — this is the structural pattern shared by Cobalt Strike Beacon (default 60s ± 20% jitter), Metasploit Meterpreter, Covenant, Sliver, and dozens of custom implants. The specific parameters vary; the architecture does not.

Part 7 — Write the detection rule

You understand the pattern well enough to write a Suricata rule that would catch it. Here is one that targets this specific implant:

Suricata rule — C2 beacon detection
alert http $HOME_NET any -> $EXTERNAL_NET any (
  msg:"Potential C2 beacon — SystemUpdater encoded UA";
  flow:established,to_server;
  http.user_agent; content:"SystemUpdater/";
  pcre:"/SystemUpdater\/[A-Za-z0-9+\/]{20,}/";
   /* fire only after 3 matches from same source in 2 minutes */
  threshold:type both, track by_src,
    count 3, seconds 120;
  sid:9000006; rev:1;
)
Q13
What does the threshold clause do, and why is it critical for beaconing detection specifically?
threshold:type both, track by_src, count 3, seconds 120 fires the alert only when this pattern matches 3 times from the same source within a 2-minute window. One matching request does not trigger it. Three does. This is the difference between a rule that fires on every pip update check and one that fires specifically on beaconing — it requires the repetition that defines the attack pattern.

Key takeaways

  • The signal is the pattern, not the packet. Each individual beacon is an informational-severity event. Fifteen beacons on a 20-second interval is a confirmed C2 channel. The content reveals almost nothing — the rhythm reveals everything.
  • You need at least three data points to see a pattern. One beacon is a coincidence. Two is suggestive. Three establishes the interval. This is why beaconing detection requires time windows, not single-event alerts.
  • Jitter defeats exact-timing rules. Rules that fire on "exactly N-second intervals" miss every real-world beacon. Threshold windows and frequency analysis — not exact timing — are how you detect jittered beacons.
  • Legitimate tools used maliciously are the hardest class of problem. You cannot block curl. You cannot alert on every curl connection. The detection is specific: this tool, this destination, this interval, this User-Agent pattern.
  • Beaconing is the waiting phase, not the action phase. While this beacon runs, the attacker is doing nothing visible. The danger is what happens when they send their first command back — Lab 7 shows what that looks like.
  • This lab sits between Labs 5 and 7 in the kill chain. The cron job from Lab 5 created this beacon. In Lab 7, the attacker uses that persistent access to exfiltrate API keys. This is the foothold that makes everything else possible.
