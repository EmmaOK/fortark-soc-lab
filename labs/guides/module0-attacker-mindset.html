How Attackers Think

20 minutes
Foundational
Read before Lab 1

Before you touch a keyboard

Every lab in this course puts you in the seat of a SOC analyst. But before you can defend against an attacker, you need to understand how an attacker thinks.

Not the Hollywood version — the hoodie, the green terminal, the dramatic music. The real version, which is far more mundane and far more dangerous.

Attackers are running a business

The single most important thing to understand about modern cybercrime is this:

Most attacks are automated. Most attackers are not sitting at a keyboard. Most attacks are not personal.

When a server gets hit with an SSH brute force attack at 3am, there is almost certainly no human watching it happen. There is a script — running on a compromised server somewhere, scanning millions of IP addresses, trying passwords from a list of 14 million real-world leaked credentials, logging any success, and moving on.

This changes how you think about detection. You are not trying to outwit a genius. You are trying to catch a pattern.

What attackers actually want

Attackers break into systems for a small number of reasons:

1. Data

Customer records. Employee credentials. Intellectual property. Health records. Financial data. All of it has a market price. A database of 100,000 email addresses and password hashes sells for hundreds of dollars on dark web markets. Credit card numbers, significantly more.

2. Access to sell

A compromised server with persistent access — a "shell" — can be sold. Criminal groups buy access to corporate networks and then deploy ransomware, conduct espionage, or use it as a launchpad to attack other targets on the same internal network.

3. Compute

Cryptocurrency mining uses significant processing power. A compromised server quietly mining cryptocurrency costs the victim in electricity and cloud bills while generating income for the attacker. Often goes undetected for months.

4. Ransomware deployment

Get access → move laterally across the network → encrypt everything → demand payment. The average ransomware payment in 2024 was over $1 million. This is not opportunistic crime. It is organised, patient, and methodical.

The attacker in Lab 1 wants credentials. They are trying to log into a Linux server via SSH. If they succeed, they have a foothold inside the network — from which they can do any of the above.

Why SSH is a target

SSH (Secure Shell) is how system administrators log into Linux servers remotely. Almost every Linux server in the world runs an SSH service. If you can log in via SSH, you have command-line access to the entire machine.

For an attacker, SSH is an attractive target because:

  • It is everywhere. Every Linux server, every cloud instance, every network device running Linux exposes SSH.
  • It uses passwords. Many organisations never disable password authentication and rely on people choosing strong passwords. People do not always choose strong passwords.
  • A successful login looks legitimate. SSH logins are normal administrative activity. A successful brute force login generates the same log entry as a legitimate administrator logging in.

The last point is critical and explains why rule 40112 — the "brute force followed by successful login" rule — matters so much. The successful login alone does not tell you anything unusual happened. Only by correlating it with the preceding failures does the picture become clear.

The attack chain

Attackers rarely do one thing. A brute force attack is almost never the end goal — it is the beginning. Once inside, a typical attacker moves through a predictable sequence:

Reconnaissance    →    discover the target exists, what services are running
Initial Access    →    brute force SSH, exploit a vulnerability, phish a user
Discovery         →    what machine is this? who else is here? what can I reach?
Credential Access →    steal passwords, SSH keys, API tokens stored on the machine
Lateral Movement  →    use stolen credentials to reach other machines on the network
Collection        →    gather files, databases, anything valuable
Exfiltration      →    send it all out over HTTPS to a server the attacker controls

This sequence has a name: the attack lifecycle or kill chain. You will map real attacks to this chain in every lab in this course. Lab 1 covers the first two stages. Lab 2 covers everything after.

The economics of scale

Here is a number that makes this concrete: there are over 20 million SSH brute force attacks per day across the public internet, detected by honeypot networks.

These attacks are not targeted at you specifically. They scan entire IP address ranges — every address on the internet — looking for anything that responds on port 22 (the default SSH port). When they find one, they try passwords. When they fail, they move on. When they succeed, they log the credential and continue.

The attack you will launch in Lab 1 against your own target is a single-threaded, deliberate, manual version of something that runs at a scale of millions of attempts per hour across the global internet.

The target you are attacking is publicly visible because its SSH port is exposed. Real servers in real organisations work the same way. The only things standing between them and a successful brute force are: strong passwords, MFA, and the SOC analysts watching the alerts.

What this means for you as an analyst

When you see an SSH brute force alert, your first question should never be "who is this hacker?" Your first questions should be:

Did they get in? — Everything else depends on this answer.

Is the targeted account real? — An "invalid user" alert means they do not even have a valid username yet. A "failed password" alert means they do — and they just need to find the right password.

Is this the only target? — A single IP hitting one server is routine noise. The same IP hitting 50 servers simultaneously is a coordinated attack requiring a different response.

These are the questions Lab 1 is designed to teach you to answer — fast, under pressure, with the evidence in front of you.

Key concepts from this module

  • Attacks are automated and impersonal — you are detecting a pattern, not outwitting a person
  • Attackers want data, access, compute, or ransom — SSH access is a means to those ends
  • The attack lifecycle is predictable — initial access is just the beginning
  • SSH is a universal target because it provides full command-line access to any Linux machine
  • The successful login alone looks legitimate — correlation with prior failures is what reveals the breach
