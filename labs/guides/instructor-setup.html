One-time setup

First-time platform deployment

Do this once per machine. Takes 20–30 minutes on a good internet connection.

1
Clone and run setup.sh
git clone https://github.com/EmmaOK/fortark-soc-lab.git
cd fortark-soc-lab
bash setup.sh

The script checks Docker, sets RAM, generates TLS certs, creates .env, and starts all containers. Takes 5–10 minutes. On first run, image pulls add another 10–15 minutes.

Windows: Run .\setup.ps1 from an admin PowerShell session in WSL2. Use docker-compose.win.yml (Suricata is disabled automatically).

2
Wait 5 minutes, then run post-start manual steps

OpenSearch and Cassandra (TheHive's DB) need time to initialise. After 5 minutes:

  • Open http://localhost/thehive/ → login as admin@thehive.local / secret
  • Admin → Organisations → Create org named SOC
  • Create users student1@thehive.local through student5@thehive.local (role: analyst, org: SOC, password: FortarkLabs1!)
  • Also create shuffle@soc.local (role: analyst, org: SOC, same password)
  • Log in as shuffle@soc.local → Profile → API Keys → Create → copy the key
  • Add to .env: THEHIVE_API_KEY=<paste key>
3
Set up Cortex
  • Open http://localhost/cortex/ → create admin account on first visit
  • Admin → Organisations → Create SOC
  • Create user thehive@cortex.local (role: read, analyze) in SOC org
  • Log in as that user → API Keys → Create → copy
  • Add to .env: CORTEX_THEHIVE_API_KEY=<paste key>
  • Restart TheHive: docker compose restart soc-thehive
4
Enable Cortex analysers and enroll Wazuh agents
bash platform/scripts/configure-cortex-analyzers.sh
bash platform/scripts/enroll-wazuh-agent.sh

The first script enables AbuseIPDB, VirusTotal, and Shodan analysers. The second installs and enrolls the Wazuh agent on every s[N]-lab-target container. Both require .env to have the API keys set (step 3).

5
Deploy lab scripts to all target containers
for N in 1 2 3 4 5; do
  for SCRIPT in labs/scripts/lab*.sh; do
    docker cp "$SCRIPT" "s${N}-lab-target:/tmp/$(basename $SCRIPT)"
    docker exec "s${N}-lab-target" chmod +x "/tmp/$(basename $SCRIPT)"
  done
  echo "s${N}-lab-target — scripts deployed"
done

This pushes all 7 lab scripts to every target container. Run this once after initial setup and again after any script update.

6
Verify the SOAR pipeline
bash platform/scripts/verify-shuffle-py.sh

Confirms shuffle.py is in place inside the Wazuh manager container and can reach TheHive. Run this after any Wazuh manager restart — the integration file can sometimes be overwritten on restart.

Lab reference

Per-lab instructor notes

Things to know and check before running each lab. Does not replace reading the lab guide.

Beginner Lab 1 — SSH Brute Force
  • Script: /tmp/lab1-brute-force-sim.sh on soc-kali-attacker (not the target)
  • Key timing: Rule 40112 needs 6+ failures from the same IP within 2 minutes, followed by a success. The bundled wordlist has 60 decoys before password123 — do not shorten it or 40112 may not fire.
  • SOAR: Not active in Lab 1. Students triage manually; SOAR is introduced in Lab 3.
  • Check before session: docker exec soc-kali-attacker ls /tmp/lab1-brute-force-sim.sh
  • Common student error: Searching Wazuh for the wrong agent name. They must filter on s[N]-lab-target, not the Kali container.
Beginner Lab 2 — Post-Breach Investigation
  • Script: /tmp/lab2-post-exploitation-sim.sh on the student's s[N]-lab-target
  • The script runs fast (~15 seconds). Students must have Wazuh open before running it. Remind them: set time range to Last 15 minutes before triggering.
  • FIM monitoring: Wazuh's FIM must be watching /tmp. If the payload file creation events don't appear, confirm the FIM config includes /tmp.
  • Cleanup: docker exec s[N]-lab-target /tmp/lab2-post-exploitation-sim.sh --cleanup
Intermediate Lab 3 — SOAR Automation
  • Depends on: SOAR pipeline working (verify-shuffle-py.sh must pass). If the pipeline is broken, SOAR alert never appears in TheHive — students will be stuck at Part 3.
  • TheHive alert vs case: The pipeline creates an Alert, not a Case. Students must click Import/Promote to create a Case from the alert. This confuses almost every first-time cohort — address it before they start Part 4.
  • Trigger: Lab 1 script (same brute force trigger). Students run it from Kali, then switch to TheHive.
  • MITRE tags: TheHive alert should include T1110 tag automatically via shuffle.py. If tags are missing, check shuffle.py is current: bash platform/scripts/verify-shuffle-py.sh
Intermediate Lab 4 — Ransomware Detection
  • Script: /tmp/lab4-ransomware-sim.sh on s[N]-lab-target
  • FIM alert storm: 40 files renamed to .locked in rapid succession. Students should watch FIM before running the script. Set time range to Last 15 minutes first.
  • Re-run issue fixed: The script cleans up its own staging directory on each run. Re-running is safe.
  • Wazuh search: Teach students to use quoted keyword search ("STAGE:INHIBIT") in the KQL bar, not the field filter dropdown. The full_log field is a keyword type — wildcards often fail.
  • Cleanup: docker exec s[N]-lab-target /tmp/lab4-ransomware-sim.sh --cleanup
Intermediate Lab 5 — Supply Chain Attack
  • Script: /tmp/lab5-supply-chain-sim.sh on s[N]-lab-target
  • Four stages in one run: Install → Dropper → Callback → Persistence. The entire chain completes in ~30 seconds. Students must have all three Wazuh tabs open before running.
  • Suricata on Windows: Disabled. Windows students skip Tab C and use "CALLBACK:BEACON_SENT" syslog filter instead. The Windows callout in the guide handles this — no need to explain unless asked.
  • Cron persistence: The script installs a real cron entry on the target. Run cleanup before the next cohort to remove it.
  • Cleanup: docker exec s[N]-lab-target /tmp/lab5-supply-chain-sim.sh --cleanup
Advanced Lab 6 — C2 Beaconing
  • Script: /tmp/lab6-c2-beacon-sim.sh on s[N]-lab-target, run in the background (&)
  • The script runs for ~5 minutes (15 beacons × 20s). Students start it, immediately switch to Wazuh, and watch in real time. Do not let them wait for the script to finish first.
  • Kill the beacon: Students write down the PID at start. To stop early: docker exec s[N]-lab-target kill <PID>
  • Suricata on Windows: Use "BEACON:" syslog filter instead of rule.groups: suricata. The syslog lines include implant_id= (base64) for the User-Agent decode exercise.
  • Internet required: Beacon calls httpbin.org. Ensure the target container has outbound internet access.
  • Cleanup: docker exec s[N]-lab-target /tmp/lab6-c2-beacon-sim.sh --cleanup
Advanced Lab 7 — LLM Key Exfiltration (Capstone)
  • Script: /tmp/lab7-llm-key-exfil-sim.sh on s[N]-lab-target
  • Decoy credentials: The script plants fake (non-functional) API keys in /tmp/lab7-exfil/. These are the targets — no real keys are used.
  • Detection gap exercise: Part 5 asks students to explain why no standard ruleset catches this. There is no single correct answer — accept any response that correctly identifies the absence of AI-credential-specific rules in Wazuh's default ruleset.
  • Internet required: Exfil and validation stages call httpbin.org and simulate a call to api.anthropic.com.
  • Final lab: Students are expected to write the most complete triage note of the course. Allow extra time (15–20 min) for the closing summary.
  • Cleanup: docker exec s[N]-lab-target /tmp/lab7-llm-key-exfil-sim.sh --cleanup

Run the appropriate cleanup after each student session or before repeating a lab. Replace [N] with the seat number, or loop over 1–5.

Lab 3No artifacts — TheHive alert/case must be closed or deleted manually in the UI
Lab 4docker exec s[N]-lab-target /tmp/lab4-ransomware-sim.sh --cleanup

Loop cleanup across all seats

LAB=4   # set to the lab number you want to clean
for N in 1 2 3 4 5; do
  docker exec s${N}-lab-target /tmp/lab${LAB}-*.sh --cleanup 2>/dev/null || \
    echo "s${N}: no cleanup needed"
done
TheHive cases and alerts are not cleaned by the lab scripts. They persist in the database until manually deleted or the full stack is reset. For a new cohort, either delete cases in the UI (Admin → Cases) or perform a full cohort reset below.
New cohort

Full cohort reset

Wipes all alert history, Wazuh events, TheHive cases, and container state. Do this between cohort groups, not between individual students.

Destructive. This deletes all Wazuh events, TheHive cases, Cortex analysis results, and target container state. You cannot undo it. Make sure you have any triage notes or reports you want to keep before running. 
# Stop the stack and delete all volumes (Wazuh indices, TheHive DB, etc.)
docker compose down -v

# Rebuild and restart — re-runs container initialisation from scratch
bash setup.sh

After setup.sh completes, repeat steps 2–6 of the First-time deployment section above to re-create TheHive accounts, re-link Cortex, and re-deploy scripts.

During a session

Monitoring students in real time

Check all containers are healthy

watch -n 30 'docker ps --format "table {{.Names}}\t{{.Status}}\t{{.RunningFor}}"'

See what a specific student's target is doing

docker exec s[N]-lab-target tail -f /var/log/syslog

Check if a student's Wazuh agent is active

docker exec soc-wazuh-manager /var/ossec/bin/agent_control -l

Verify the SOAR pipeline is intact

bash platform/scripts/verify-shuffle-py.sh

Watch TheHive for incoming alerts

Log in to TheHive as admin@thehive.local and view the Alerts list. You can see all student-generated alerts in real time from the admin account.

When things break

Troubleshooting

SymptomFix
Wazuh dashboard shows "no index pattern" or blank Wait 3–5 more minutes for OpenSearch indices to initialise. Refresh the page. If still blank after 10 min: docker restart soc-opensearch soc-wazuh-dashboard
OpenSearch crashes immediately on Linux sudo sysctl -w vm.max_map_count=262144. Make permanent by adding to /etc/sysctl.conf.
OpenSearch crashes on Windows WSL2 Add kernelCommandLine = sysctl.vm.max_map_count=262144 to %UserProfile%\.wslconfig, then restart WSL2: wsl --shutdown
Rule 40112 never fires in Lab 1/3 The wordlist needs 6+ failures before the success. Use the bundled lab1-brute-force-sim.sh — do not shorten the wordlist. Also check the agent is active in Wazuh.
TheHive alert never appears (Lab 3) 1. Confirm rule 40112 fired in Wazuh. 2. Run verify-shuffle-py.sh. 3. Check docker logs soc-wazuh-manager | grep shuffle for errors. Most common cause: stale shuffle.py missing the THEHIVE_API_KEY.
TheHive shows "Cortex unreachable" Check CORTEX_THEHIVE_API_KEY in .env. Restart: docker compose restart soc-thehive
Wazuh agent shows Disconnected docker restart s[N]-lab-target. Agent re-connects automatically within 60 seconds.
Lab script gives "same file" error (Lab 4) Run docker exec s[N]-lab-target /tmp/lab4-ransomware-sim.sh --cleanup to clear the previous run's .locked files, then re-run the script. (Fixed in current version — only affects old deployments.)
Suricata alerts missing (macOS) Suricata uses pcap on macOS via the Docker bridge. If no Suricata events appear, check: docker logs soc-suricata. On macOS it is normal for Suricata to miss some inter-container traffic.
shuffle.py overwritten after Wazuh restart Run bash platform/scripts/verify-shuffle-py.sh — it detects and restores the file automatically.
Kali terminal page hangs / no prompt docker restart soc-kali-attacker. Reload the browser tab after ~15 seconds.
Wazuh search shows no results for a lab stage 1. Check time range is set to Last 15 minutes (not Today). 2. Use quoted keyword search in the KQL bar, not the field filter dropdown. 3. Confirm the syslog rule matched — check docker exec soc-wazuh-manager tail -f /var/ossec/logs/alerts/alerts.json | grep LAB.
Windows deployment

Windows / WSL2 deployment

The platform runs on Windows via WSL2. Two differences from macOS:

  • Use setup.ps1 not setup.sh. Run from an admin PowerShell session.
  • Suricata is disabled in docker-compose.win.yml. WSL2 cannot capture raw packets across Docker bridge interfaces. Labs 5, 6, and 7 include inline syslog alternatives for Windows students — no guide changes needed.
# Windows — start the stack
docker compose -f docker-compose.yml -f docker-compose.win.yml up -d

# Windows — stop
docker compose -f docker-compose.yml -f docker-compose.win.yml down
Students on Windows see the Windows users: callout boxes in Labs 5–7 automatically. They use Wazuh syslog filters instead of Suricata network alerts. All five rubric categories are achievable without Suricata.
