What is post-breach investigation?

Three terms you need to understand first
Post-Breach
"Post" means after. Post-breach means after the attacker has already gotten in. The question shifts from "how do I stop them" to "what did they do, when, and what did they take." Most security training focuses on prevention. This lab focuses on what happens when prevention fails — because it does fail, regularly.
IOC — Indicator of Compromise
Any piece of evidence that a system was compromised — an IP address the attacker connected from, a file they created, a command they ran, a domain they contacted. IOCs are the footprints attackers leave behind. Collecting them is how you prove what happened and prevent the same attacker from hitting the next target.
Enrichment
Looking up an IOC in external threat intelligence databases to learn more about it. If you find an IP address in your logs, enrichment means asking: "Has anyone else seen this IP in an attack? Is it on a known-bad list?" Enrichment turns a raw IP address into context — and context turns an alert into a conclusion.

Analogy: Think of it like a crime scene investigation. The attacker has left the building. The detective arrives and starts looking for evidence — fingerprints (file hashes), footprints (network logs), witnesses (syslog entries). Each piece of evidence is an IOC. Running that evidence through a database is enrichment. The goal: reconstruct exactly what the criminal did, even after they are gone.

The scenario

The brute force from Lab 1 succeeded. The attacker is now inside your server, logged in as victim.

What they do next happens fast — under 30 seconds from first keystroke to data leaving the building. Most organisations do not detect this phase at all. Their alerts fire on the brute force but miss everything that comes after.

Your job in this lab is different. You are not going to look at the brute force again. You are going to investigate what happened after the attacker got in — reconstruct their moves step by step from the evidence they left behind, build a complete incident timeline, and document it in a case management system the way a real SOC team would.

This is where the real investigation work lives.

How post-exploitation detection works

An attacker inside a system leaves traces in three different places. Each requires a different detection tool:

What the attacker does Detection tool Alert type
Runs commands (id, uname, ps) Wazuh Syslog alert (auth.log)
Reads sensitive files (/etc/passwd, SSH keys) Wazuh FIM File integrity monitoring alert
Creates a file in /tmp/ Wazuh FIM File integrity monitoring alert
Sends data out over HTTPS Suricata Network intrusion detection alert
No single tool sees everything. Your job is to collect the fragments from each source and assemble them into a single coherent story. This is what incident investigation actually is.

Part 2 — Hunt the 5 stages

The simulation ran 5 distinct attack stages. Your job now is to find evidence of each one in the SOC platform — as if you had no idea what the script did.

Work through each stage in order. Use the filters provided but read the actual alert content to understand what you are looking at.

1
System Discovery
T1082 T1033 Wazuh syslog

The first thing any attacker does after getting in is figure out where they are. They run basic commands to learn about the machine: what OS is running, who the users are, what processes are active.

Wazuh filter
agent.name: s[N]-lab-target AND full_log: *DISCOVERY*
Q1
What commands did the attacker run? (Look at the log messages — they list the commands explicitly)
Q2
Why would an attacker run id and who immediately after logging in?
These commands answer the two most important questions for an attacker: "How much power do I have?" (id reveals if they have root) and "Is anyone watching?" (who shows if other users are logged in). This reconnaissance takes under 3 seconds and happens on every intrusion.
2
Credential File Access
T1552.001 Wazuh FIM

The attacker then goes hunting for credentials and access keys stored on the machine.

Wazuh filter
agent.name: s[N]-lab-target AND rule.groups: syscheck

Look for FIM (File Integrity Monitoring) alerts. Wazuh watches certain files and directories and fires an alert any time they are accessed or changed.

Q3
Which sensitive file was accessed? (Look for the syscheck.path field in the alert)
Q4
Why is /etc/passwd valuable to an attacker?
The /etc/passwd file lists every user account on the system — usernames, home directories, and shell types. Combined with /etc/shadow (the password hash file), an attacker can attempt to crack passwords offline. Even without the hashes, the username list helps them target other accounts on the network.
3
Network Reconnaissance
T1049 T1016 Wazuh syslog

Next the attacker maps the network — what connections are open, what services are running, what other machines can they reach from here.

Wazuh filter
agent.name: s[N]-lab-target AND full_log: *RECON*
Q5
What network commands did the attacker run?
Q6
Why does an attacker care about the network configuration of a machine they have already compromised?
Getting into one machine is rarely the end goal. The attacker is looking for paths to move laterally — other servers they can reach from here, internal services now accessible because they are inside the network, credentials they can reuse. The network recon tells them what their next move is.
4
Data Staging
T1074.001 Wazuh FIM

Before exfiltrating data, attackers bundle everything they collected into a single compressed archive. This is called staging.

Wazuh filter
agent.name: s[N]-lab-target AND rule.groups: syscheck

Look for a FIM alert showing a new file created in /tmp/.

Q7
What is the full path of the staged file?
Q8
Why do attackers compress and archive data before exfiltrating it?
Sending one compressed file is faster, harder to detect, and easier to encrypt than sending multiple individual files. It also means the exfiltration looks like a single outbound connection rather than a stream of file reads — making it harder to correlate with the earlier credential access.
5
Exfiltration
T1041 Suricata

The final stage: the staged archive leaves the building. Suricata caught the outbound HTTP POST at the network layer.

Wazuh filter
data.alert.signature_id: 9000010

This alert comes from Suricata, the network intrusion detection system. Unlike Wazuh (which watches what happens on the host), Suricata watches network traffic. It caught the outbound HTTP POST carrying the loot archive.

Note on internet noise: httpbin.org is a public test endpoint — you may see alerts from other containers or background traffic. Filter precisely using data.alert.signature_id: 9000010 to isolate only the lab exfil alert from your target.

Once you find the alert, expand it and look at these fields:

Field What it tells you
data.dest_ipDestination IP the data was sent to
data.dest_portDestination port (80 = plain HTTP)
data.app_protoApplication protocol detected by Suricata
data.alert.metadata.flowbitsFlowbits set by ET rules — e.g. ET.formdata means a multipart form-data body (file upload) was detected
data.directionTraffic direction — should be to_server (outbound)
Q9
What is the destination IP of the exfiltration? (Read it from data.dest_ip)
Q10
What does the ET.formdata flowbit tell you about how the data was sent?
The ET.formdata flowbit means the POST body was a multipart form-data upload — the same format used to upload files through a web form. Attackers use this to disguise data exfiltration as normal web traffic. The data moved over plain HTTP on port 80, making it fully visible in transit but blending in with ordinary web requests.

Q13
Is the destination flagged as malicious by any of the three tools?
The lesson: clean infrastructure is the attacker's strategy.

The destination is httpbin.org — a completely legitimate developer testing service. AbuseIPDB: zero reports. VirusTotal: clean across all 90+ engines. Shodan: standard web server, reputable CDN owner.

Attackers deliberately choose destinations like this. httpbin.org, Pastebin, GitHub, Google Drive, Dropbox, OneDrive — trusted by every corporate firewall, zero alerts from any IP reputation system. Blocking them would break legitimate developer and business workflows.

This is why behavioural detection matters more than blocklists. Suricata fired not because httpbin.org is malicious. It fired because an outbound POST with a file attachment from a server that has no business sending file uploads is anomalous behaviour, regardless of the destination's reputation.

A clean enrichment result is not evidence of innocence. It is evidence of a competent attacker who chose their infrastructure carefully.

Key takeaways

  • Most breaches are detected in post-exploitation, not initial access. The brute force is loud and obvious. What the attacker does after getting in is quieter and harder to catch — but it is where the actual damage happens.
  • Post-exploitation leaves traces in three different places. Host logs (Wazuh), file access (FIM), and network traffic (Suricata). No single tool sees the full picture. Correlation is the analyst's core skill.
  • Speed makes human response nearly impossible. Login to data exfiltration in under 30 seconds. Automated detection and pre-built response playbooks are not optional — they are the only way to respond in time.
  • IP reputation tools miss clean infrastructure. httpbin.org is not malicious. Attackers know this and deliberately use legitimate services. Behaviour-based detection catches what blocklists cannot.
  • The TheHive case is the product. The alerts are just raw material. The documented, structured, properly tagged incident case with a clear verdict and complete IOC list is what actually serves the organisation.
  • Claude turns fragments into narrative. An analyst with 10 alert timestamps and a list of file paths can produce a board-level incident briefing in 30 seconds. That is the leverage that makes AI genuinely useful in a SOC.
