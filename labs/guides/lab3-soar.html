What is SOAR?

Two terms you need to understand first
SOAR — Security Orchestration, Automation, and Response
Tools that connect your security products together and automate repetitive response tasks. On its own, Wazuh detects a brute force and fires an alert that sits in a queue until someone opens it — hours later. SOAR changes this: a pre-written playbook runs automatically the moment the alert fires. The case is created, the IP is logged, the analyst is notified. No human involvement. No delay.
Playbook
A pre-written sequence of steps that runs automatically in response to a trigger. "If a brute force alert fires, create a TheHive case, tag it with the attacker's IP, set severity to High, and notify the on-call analyst." That sequence, codified into a workflow, runs the same way every time, in under 3 seconds, without anyone having to remember the steps.

Analogy: Think of a hospital's cardiac emergency system. When a patient's heart monitor flatlines, it does not wait for a nurse to notice the screen. It triggers an alarm, pages the crash team, and unlocks the crash cart — automatically, in the first 10 seconds. The team arrives to a room that is already prepared. SOAR does the same thing for a SOC: the analyst arrives to a case that is already documented, timestamped, and tagged.

The scenario

In Lab 1 you ran a brute force attack. You found the alert manually.

In Lab 2 you investigated what happened after the attacker got in. You built the timeline manually.

Now imagine you are running a SOC for 500 servers. A new brute force attempt hits every 3 minutes. You cannot investigate each one by hand — you will be buried by the time any analyst opens the first ticket.

This is the problem SOAR solves.

SOAR (Security Orchestration, Automation and Response) intercepts the Wazuh alert automatically and runs a response playbook — creating a TheHive incident case, tagging it with MITRE ATT&CK techniques, and logging the attacker's IP, all within 3 seconds of the first brute force packet hitting your server. No human involvement. No delay. The case is already waiting when an analyst opens their dashboard in the morning.

Part 1 — Tour the automation pipeline before you trigger it

Before running the attack, explore the components that will respond to it. Understanding each piece makes the automation feel real rather than magical.

Step 1 — Look at the integration script

This lab uses a direct integration: Wazuh calls a Python script (shuffle.py) that sends the alert straight to TheHive — no Shuffle worker involved. You can browse Shuffle at https://192.168.6.213:3001 to explore its workflow UI, but the SOAR automation in this lab runs through the script, not through a Shuffle workflow.

A
In ossec.conf — the Wazuh manager config has an integration block that says: fire on any alert of level 7 or higher, call the built-in shuffle wrapper, and pass the full alert as JSON.
B
The shuffle.py script reads the alert JSON, checks whether it is rule 40112 (brute force + successful login), and if so, makes an HTTP POST directly to http://soc-thehive:9000/api/v1/alert with a structured payload containing the agent name, attacker IP, severity, and MITRE tags.
Q1
What does the shuffle.py script check before creating a TheHive alert? (Why doesn't every level-7 alert create a case?)
The script checks for rule 40112 specifically. Rule 40112 matches both a brute force attempt and a successful login — the combination that signals an actual breach. Many brute force attempts fail and are loud, routine events. Rule 40112 is the quiet, dangerous one: it means the attacker found the password. That is the event worth automatic case creation.

Q7
What is the difference between an alert and a case in TheHive?
An alert is an automated notification — a raw signal that something may have happened. It is not yet confirmed, not yet assigned, and not yet under active investigation.

A case is a confirmed incident under investigation — assigned analyst, structured timeline, tasks, and a verdict. Promoting an alert to a case is the analyst's judgement call that says: "This is real. We are working it."

Q8
In a SOC monitoring 500 endpoints with 200 alerts per day, how much analyst time per day does this automation save?
Rough math: creating a TheHive case manually takes ~5 minutes for a well-practiced analyst. 200 alerts × 5 minutes = 1,000 minutes = 16+ analyst-hours per day, just on ticket creation. SOAR doesn't replace the investigation — it eliminates the administrative overhead so analysts can spend those hours on actual analysis.
Q9
What are the risks of fully automated case creation — what could go wrong?
Two main risks:

(1) False positives — if the detection logic is too broad, SOAR creates cases for legitimate events, flooding the queue and training analysts to ignore alerts.

(2) Automated response overreach — if the playbook goes further than case creation (e.g., automatically blocking IPs, killing processes, isolating hosts), a false positive becomes an outage. This is why most SOCs automate case creation and enrichment, but keep containment actions as human-approved steps.

Part 6 — Close the case

In a real incident, a case is only closed after the full incident response lifecycle is complete. Closing early — before the attacker is out and the system is clean — means you declared victory while the breach was still active.

The correct order is Contain → Eradicate → Recover → Document → Close. Work through the full checklist below.

Key takeaways

  • SOAR is not magic — it is plumbing. The automation pipeline is just a sequence of API calls: Wazuh calls Shuffle, Shuffle calls TheHive. Understanding each connection means you can troubleshoot when it breaks.
  • Automation is only as good as the detection. Shuffle created the case because Wazuh fired rule 40112. If the detection rule was wrong, every case would be a false positive. SOAR amplifies both good and bad detection logic.
  • The case is still yours to close. Automation handles the administrative layer — ticket creation, tagging, enrichment. The analyst judgement — is this real? what is the impact? what do we do next? — stays with the human. Automation gives you more time to think, not permission to stop thinking.
  • Speed changes the game. Lab 1 showed you the attack. Lab 2 showed you the aftermath. Lab 3 shows you that by the time a real analyst would have opened the Wazuh dashboard, the case was already created, tagged, and waiting. In incidents where minutes matter, that gap is the difference between containment and breach.
  • Alert fatigue is real. A SOC that creates a case for every single Wazuh alert would be unusable within a week. The filter — rule 40112 specifically, not every level-7 alert — is a deliberate tuning decision. Getting that filter right is one of the hardest ongoing problems in detection engineering.
